## Step 2 ‚Äî Build check (conditional)

Look up the project's build check in `CLAUDE.md ¬ßCommands`. Run whatever is defined there (e.g. `npm run build`, `cargo build`, `make`, `supabase db reset`, etc. ‚Äî whatever the project considers a build verification).

If `CLAUDE.md ¬ßCommands` defines no build step (e.g. a markdown-only repo, a domain project with no software build), skip this step silently ‚Äî no noise.

If the build fails, fix errors before proceeding. Do not commit broken code.

## Step 3 ‚Äî Commit and push branch

Stage and commit all uncommitted changes with `git add -A`. Write a commit message that:
- Starts with the phase/task (e.g. "Phase 5.5 ‚Äî")
- Summarizes what was done in plain English
- Ends with `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`

If there is nothing to commit, skip the commit and say so.

Then push based on the current branch ‚Äî **do not open a PR yet**:

**On `main` (DEC-005 solo flow):**
Run `git push origin main`. Push unconditionally ‚Äî catches any earlier-session unpushed commits so the stop hook stays quiet through `/kill-this` ‚Üí `/its-dead`. Skip Steps 4 and 5 ‚Äî no PR needed. Go straight to Step 6.

**On a `task/*`, `claude/<slug>`, or feature branch (PR flow):**
Capture: `BRANCH=$(git branch --show-current)` and `SUBJECT=$(git log -1 --format=%s)`.
Push the branch: `git push -u origin $BRANCH`. Do not open a PR yet ‚Äî code review runs first.

## Step 4 ‚Äî Code review

Run the @code-review agent against HEAD (`git diff HEAD~1`). Wait for it to complete. Capture the findings ‚Äî you'll need them for the PR body and the session log draft.

## Step 5 ‚Äî Open PR (feature branches only)

Now open the PR with code review findings included in the body.

Check for an existing open PR first: `gh pr view $BRANCH --json url,state -q '.state' 2>/dev/null`. If output is `OPEN`, capture the URL with `gh pr view $BRANCH --json url -q '.url'` and skip to surfacing the URL ‚Äî don't open a duplicate.

Otherwise: `gh pr create --base main --head $BRANCH --title "$SUBJECT" --body "<auto-generated body>"`. Body must include all four sections:

**## Summary**
One-line description of what changed and why.

**## Files changed**
Bulleted list from `git diff --name-only main..HEAD`.

**## Code review**
Paste the findings from Step 4 here. If clean, say "Clean bill of health."

**## Test plan**
Checkbox list tailored to what actually changed ‚Äî never skip this. Projects should customize; example for a Next.js + Supabase project:
- [ ] Playwright tests run for changed files?
- [ ] Migration needed? ‚Üí `supabase db push` on remote
- [ ] RLS change? ‚Üí `supabase test db` passes
- [ ] UI change? ‚Üí verified at 375px

Capture the returned PR URL. Surface it in your response and note it in the draft session log entry's `Context` section.

**Fallback if `gh pr create` fails for any reason** (unavailable, unauthenticated, missing `pull_request:write` scope, org policy, network, etc.):
Push already succeeded, so the branch is on the remote. Tell the user to open the PR manually. Note the missing PR in the draft log's Context.
